# Whitepaper 模板（行业白皮书 / 技术白皮书 / Vendor 报告）

> 4 类之一：`--type whitepaper`。白皮书 PDF → 中文结构化总结（无图抽取，概念用 mermaid / 表格 / ASCII 示意图）。
>
> 适用对象：行业研究报告、技术白皮书（vendor 视角）、市场分析、调研报告、咨询报告、政策报告等。
>
> **章节顺序 SSOT**：本文件正文 + SKILL.md §whitepaper 输出小节；改一处必须同步另一处。

## 输出与字符数纪律

**字符数目标：≤ 3500**（含中英文/空格/标点；这是**目标**不是上限）。

**单模式**：whitepaper 不分 quick / full。

**图表处理**：走 `_base.md` §4.2——不抽原始图，概念用 mermaid / 表格 / ASCII 在正文里直接画。

## 模板

````text
你是一位行业研究分析师。请基于这篇 PDF 白皮书（含行业研究报告 / vendor 技术白皮书 /
市场分析 / 调研报告 / 政策报告），用**中文**输出**结构化总结**，严格按以下章节顺序与命名
（不存在的章节可省略）。
**每个章节标题一律用 `##`（二级标题），不要降级成 `###` 或更低**。

## 章节骨架

```markdown
| **白皮书标题** | **发布方** | **主题** |
|-------|-------|-------|
| 白皮书全名（中文或英文） | 厂商 / 机构 / 咨询公司 | 行业 / 主题关键词 |

* **发布日期**：<发布日期，如能识别>
* **目标读者**：<决策者 / 工程师 / 投资人 / 政策制定者 / ...>
* **发布立场**：<vendor 自家方案 / 行业中立研究 / 学术综述 / 政府报告 / 咨询报告>
```

**【必填项强调】**：

- 三列至少**二选一必填**——发布方与主题识别不出时**整段省略表格**，
  直接进入"行业背景"
- 发布立场是白皮书**核心信号**——vendor 自家方案 vs 行业中立，读者期望值完全不同；
  无法判断时**省**该字段，**不编造**
- 目标读者不要泛泛写"业内人士"，从文档摘要 / 引言推断

## 行业背景

* **行业 / 领域现状**（一句话定位：什么领域、什么时间窗、什么地理范围）
* **关键趋势 / 驱动因素**（**2-3 条最关键的**，避免罗列所有 PEST 分析维度）
* **市场规模 / 增长数据**（若有具体数字则保留，避免"显著增长"这类空话）

## 挑战与机遇

* **现有挑战 / 痛点**（**3-5 条**，按重要性排序；每条 1-2 句话）
* **机遇 / 增长点**（与挑战呼应——挑战的反面就是机遇）
* **为什么现在是关键时间点**（如文档有"窗口期 / 拐点"叙事则保留；无则省略）

## 解决方案 / 核心主张

> **这是白皮书的"灵魂"**——vendor 白皮书这里是自家方案；行业报告这里是
> 政策建议 / 行业共识 / 未来路径。

* **核心主张**（一句话讲清"作者想让读者相信什么"）
* **方法 / 框架 / 方案要点**（**3-6 条**，每条 1-2 句话讲清"是什么 + 为什么有效"）
* **关键技术 / 数据 / 论据**（支撑主张的核心证据，按 bullet 列出关键数字 / 案例 / 研究结论）
* **架构 / 流程 / 框架图**（如文档有总览图，用 ` ```mermaidjs ` block 画；不要照搬文档原图）

## 客户案例 / 实证

* **典型案例**（2-4 条，每条 1-2 句话："客户 + 场景 + 关键收益 + 数据"）
* **关键数据汇总**（若有客户收益数据则转 markdown table，不要罗列所有 case）
* **行业 benchmark / 对比**（如文档给出 vs 同业的对比数据则保留）

## 商业价值 / 落地建议

* **ROI / 商业价值**（若文档有具体数字则保留——避免"提升效率"这类空话）
* **落地路径 / 实施建议**（3-5 条，按"评估 → 试点 → 推广"或类似阶段组织）
* **风险与对策**（vendor 白皮书通常会说"如何选择合适方案"；行业报告通常说"政策风险 / 监管风险"）
* **未来展望**（**1-2 句话**展望 1-3 年趋势；不要展开成长文）

## 附录（可选）

* **关键术语表**（若有）：markdown 表格（术语 / 缩写 / 含义）
* **参考链接**（文档里**明确给出**的；按 "原始报告 / 数据来源 / 在线 Demo / 视频 / 博客"分组；
  绝不凭自己知识补 URL；文档完全未给链接时整段省略）
* **关于发布方**（1-2 句话，简述发布方在该领域的资历 / 立场）——仅当文档有专门"About Us"
  章节时输出
````

## 写作样例

```markdown
| **白皮书标题** | **发布方** | **主题** |
|-------|-------|-------|
| The State of AI Infrastructure 2026 | Gartner | AI Infra / LLM serving |

* **发布日期**：2026-02-10
* **目标读者**：CTO / 基础设施架构师 / 投资人
* **发布立场**：行业中立研究

## 行业背景

* LLM serving 基础设施市场 2025 年规模约 80 亿美元，预计 2028 年达 420 亿美元（CAGR 70%+）。
* 三大驱动：**模型规模扩张**（单卡推理已成少数）/ **推理成本压力**（token 成本压缩 10×）/ **多模态兴起**（GPU 利用率优化空间大）。

## 挑战与机遇

* **GPU 短缺与利用率不均并存**：头部厂商把 A100/H100 利用率压到 70%+，但中长尾客户 < 30%
* **延迟 vs 成本 trade-off 极端化**：P99 latency < 100ms 与 < $1/M tokens 难以兼得
* **跨厂商可移植性差**：CUDA 生态锁定，ROCm / 国产 NPU 成熟度仍不足

## 解决方案 / 核心主张

* **核心主张**：未来 3 年 LLM serving 基础设施的核心矛盾从"卡够不够"转向"调度够不够聪明"
* **方法 / 框架**：
  * **异构调度**（GPU + NPU + CPU 协同），目标利用率 85%+
  * **推测解码（speculative decoding）**规模化部署
  * **KV cache 跨请求复用**（prompt cache）作为一级架构

## 商业价值 / 落地建议

* **落地路径**：
  * 阶段 1（0-6 月）：内部推理网关 + 缓存复用 → 成本 -30%
  * 阶段 2（6-12 月）：异构调度 + 推测解码 → 延迟 P99 -50%
  * 阶段 3（12-24 月）：跨厂商可移植层 + 多模态协同
```
