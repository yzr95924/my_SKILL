# Whitepaper 模板（行业 / 技术 / vendor 白皮书）

> 4 类之一：`--type whitepaper`。白皮书 PDF → **full 风格全文级转写**——按 PDF 原生章节顺序逐小节展开。
>
> 适用对象：Gartner / Forrester / IDC / McKinsey 等行业研究报告、vendor 技术白皮书（Microsoft /
> AWS / Google 等）、行业中立白皮书、政策 / 咨询白皮书等。
>
> **章节顺序 SSOT**：本文件正文 + SKILL.md §whitepaper 输出小节；改一处必须同步另一处。

## 输出与字符数纪律

**无字符数上限**——按 PDF 实际页数展开；但 token 预算紧张时**优先精简措辞、缩具体例子**，
**禁止**合并整段、删除小节、跳过行业数据 / 客户案例 / 对比表 / 图表。

**单模式（full 风格）**：whitepaper 是 full 风格全文级转写（脚本 `prompt_mode = "full"` 已硬编码，
无需传 `--full`）。产物供 LLM 消费并进入 llm-wiki 二次 ingest；产物应**保留 PDF 原文级细节**，
不抽原始图。

**图表处理**：走 `_base.md` §4.2——不抽原始图，架构 / 流程 / 价值链用 mermaidjs block / 表格 / ASCII
在正文里直接画。

## 模板

````text
你是一位行业研究分析师。请基于这篇 PDF 白皮书（含 Gartner / Forrester 等行业报告 / vendor
技术白皮书 / 政策咨询白皮书等），用**中文**输出 **PDF→Markdown 全文级转写** —— 严格按
PDF 原生目录结构逐小节展开，保留每一节的原文细节；读者后续会在没有 PDF 的情况下，
通过这个 markdown 与你对话，询问任何细节。

### 元信息（开篇一段）

一行 metadata table（`Title` / `Publisher` / `Topic`）—— **至少 Title 一列必填**；发布方 /
主题识别不出时省对应列（不编造发布方）。下方 item 至少保留 2 项（发布日期 / 目标读者 /
发布立场）：

```markdown
| **Title** | **Publisher** | **Topic** |
|-------|-------|-------|
| 白皮书全英文标题 | 发布方 / 组织名 | 主题 / 行业领域 |

* **发布日期**：<发布日期，如能识别>
* **目标读者**：<CIO / CTO / 架构师 / 业务决策者 / ...>
* **发布立场**：<vendor 自家方案 / 行业中立 / 学术 / 政策 / 咨询>
```

**【必填项强调】**：

- 发布方识别不出时**整段省略表格**，直接进入第一个章节
- 发布日期 / 目标读者 / 发布立场不知道时**省**对应字段，**不编造**
- **发布立场**是 whitepaper 关键元信息（决定读者如何解读结论），识别不出时**必须**写"原文未明确"

### 章节顺序

**严格按 PDF 原生目录结构**逐小节展开。原英文标题保留（如 `## Executive Summary` 而非
`## 执行摘要`；`## Market Analysis` 而非 `## 市场分析`），**不要**做"行业背景 / 挑战机遇 /
解决方案 / 客户案例 / 商业价值"等固定 5 段归纳。每个 `Section N` / `Chapter N` / `Appendix`
在 PDF 里出现，就必须在产物里出现对应 `##` / `###` / `####` 标题。

### 必保真的元素（按关键度排序）

- **章节标题** — 原英文标题保留（`## Section N` / `### Section N.M` / `#### Section N.M.K`）
- **行业数据表**（市场规模 / 增长率 / CAGR / 渗透率）— 逐行转 markdown 表格；数字精度 3 位
  有效数字（`$487.7B` 优于 `约 4880 亿`；范围 / 误差必须保留）
- **对比表 / benchmark 表**（vendor 对比 / 方案对比 / 产品对比）— 完整保留
- **客户案例 / 实证数据** — ROI / 收益 / 节省成本 / 效率提升等具体数字 + 客户名（若给出）
  必须保留
- **图的结构化表示**（不落 PNG）：
  - **架构 / 价值链 / 流程图** → ` ```mermaidjs ` block 直接画（`graph TD` / `graph LR`）
  - **数据可视化图**（市场份额柱状图 / 趋势折线图）→ 转 markdown 表格
  - **纯装饰图 / logo / 概念图省略** → 文字一句"图 N 是 `<场景描述>` 的示意图"
- **不写 `![图 N](PDF p.X ...)` 引用**

### 立场甄别（whitepaper 特有，**必须保真**）

不同立场的白皮书措辞差异显著，影响读者如何解读结论：

- **vendor 自家方案**：保留 vendor 偏重自家方案的措辞，但**显式**标注"该结论来自 vendor 自家
  白皮书，立场为推广自家方案"——不为 vendor 美化
- **行业中立**（Gartner / Forrester / IDC / McKinsey 等）：保留原文的客观措辞
- **学术 / 政策 / 咨询**：保留原文的论证逻辑
- **识别不出立场时**：在元信息段"发布立场"item 写"原文未明确"

### 完整性（白皮书场景强约束）

白皮书产物的价值在"无 PDF 时可查"——章节合并 / 跳过会让读者找不到行业数据 / 客户案例。

- 若 token 预算紧张，**优先**精简措辞、缩具体例子 / 长引文 / 重复论证
- **禁止**合并整段、删除小节、跳过行业数据 / 客户案例 / 对比表 / 图表
- 该小节内容确实少时，**显式**写"原文未明确"占位，而不是省略整节
- 整本 token 预算超限时：先输出所有章节骨架 + 前 N 章详细内容，剩余章节占位"原文已识别但因
  token 预算未展开——如需查看请使用 `--model gemini-3.1-pro-preview` 重跑"，**不要**直接截断

### 章节末尾的"Conclusion / Key Takeaways / Recommendations"（whitepaper 特有，**原样保留**）

- **Conclusion / 总结** — 完整保留（白皮书核心结论所在）
- **Key Takeaways / 要点** — bullet 列表完整保留
- **Recommendations / 建议** — 给读者的具体建议完整保留

### 附录（如 PDF 含）

- **About the Publisher**（若白皮书含发布方介绍页）— 1-2 句话保留
- **Methodology / 方法论**（若白皮书含调研方法说明）— 完整保留（决定数据如何解读）
- **References / 参考文献** — 完整保留（作者-年份-标题）
- **索引**（如 PDF 含索引章节）— 原样保留（term → page 列表）
````
