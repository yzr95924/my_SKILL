# Manual 模板（产品手册 / 用户手册 / Datasheet）

> 4 类之一：`--type manual`。产品手册 PDF → **full 风格全文级转写**——按 PDF 原生章节顺序逐小节展开。
>
> 适用对象：硬件 datasheet、软件产品手册、API 参考手册、安装运维指南、设备操作手册、SLA 文档等。
>
> **章节顺序 SSOT**：本文件正文 + SKILL.md §manual 输出小节；改一处必须同步另一处。

## 输出与字符数纪律

**无字符数上限**——按 PDF 实际页数展开；但 token 预算紧张时**优先精简措辞、缩具体例子**，
**禁止**合并整段、删除小节、跳过命令 / 参数 / 错误码 / 表格行。

**单模式（full 风格）**：manual 是 full 风格全文级转写（脚本 `prompt_mode = "full"` 已硬编码，无需传 `--full`）。
产物供 LLM 消费并进入 llm-wiki 二次 ingest；产物应**保留 PDF 原文级细节**，不抽原始图。

**图表处理**：走 `_base.md` §4.2——不抽原始图，架构 / 概念用 mermaid block / 表格 / ASCII 在正文里直接画。

## 模板

````text
你是一位产品技术写作助手。请基于这篇 PDF 产品手册（含 datasheet / 用户手册 / 安装运维指南等），
用**中文**输出 **PDF→Markdown 全文级转写** —— 严格按 PDF 原生目录结构逐小节展开，
保留每一节的原文细节；读者后续会在没有 PDF 的情况下，通过这个 markdown 与你对话，询问任何细节。

### 元信息（开篇一段）

一行 metadata table（`Title` / `Type` / `Vendor`）—— **至少 Title 一列必填**；产品名 / 类型 / 厂商
识别不出时省对应列（不编造厂商名）。下方 item 至少保留 2 项（手册版本 / 目标读者 / 关联产品）：

```markdown
| **Title** | **Type** | **Vendor** |
|-------|-------|-------|
| 产品全名（如 "NVIDIA H100 SXM" / "Kubernetes 1.30"） | 类别（GPU / 操作系统 / 数据库 / 路由器 / ...） | 厂商 / 组织名（识别不出省） |

* **手册版本 / 日期**：<手册版本号>（<发布日期，如能识别>）
* **目标读者**：<工程师 / 运维 / 终端用户 / 开发者 / ...>
* **关联产品**：<手册涉及的其它产品 / 版本>
```

**【必填项强调】**：

- 厂商与产品名识别不出时**整段省略表格**，直接进入第一个章节
- 手册版本 / 日期 / 目标读者 / 关联产品不知道时**省**对应字段，**不编造**
- 不加额外副条目（不要 `* **售价**：...` / `* **联系方式**：...` 这类与文档结构无关的字段）

### 章节顺序

**严格按 PDF 原生目录结构**逐小节展开。原英文标题保留（如 `## Hardware Specifications` 而非
`## 硬件规格`；`## Installation` 而非 `## 安装`），**不要**做"产品概述 / 关键特性 / 接口参数"等
固定 6 段归纳。每个 `Chapter N` / `Section N.M` / `Appendix` 在 PDF 里出现，就必须在产物里
出现对应 `##` / `###` / `####` 标题。

### 必保真的元素（按关键度排序）

- **章节标题** — 原英文标题保留（`## Section N` / `### Section N.M` / `#### Section N.M.K`）
- **命令清单 / API endpoint** — 完整保留，转 ` ```bash ` / ` ```python ` / ` ```yaml ` / ` ```json `
  fenced block；多参数时用 markdown 表格（参数名 / 类型 / 必选 / 说明）
- **参数 / 配置 / 环境变量表** — 逐行转 markdown 表格；数字精度 3 位有效数字（如 `3.35 TB/s`
  优于 `约 3 TB/s`；范围 / 误差必须保留）
- **错误码 / 异常表** — 逐行转 markdown 表格（错误码 / 含义 / 处置 / 严重级别）
- **产品参数表**（datasheet 关键）— 型号 / 容量 / 性能 / 接口 / 协议 / 标准 — 完整保留
- **图的结构化表示**（不落 PNG）：
  - **架构 / 模块关系图** → ` ```mermaid ` block 直接画（`graph TD` / `graph LR`）
  - **数据可视化图**（性能曲线 / benchmark）→ 转 markdown 表格
  - **纯装饰图 / logo / 概念图省略** → 文字一句"图 N 是 `<场景描述>` 的示意图"
- **不写 `![图 N](PDF p.X ...)` 引用**

### 完整性（手册场景强约束）

手册产物的价值在"无 PDF 时可查"——章节合并 / 跳过会让读者找不到命令 / 参数 / 错误码。

- 若 token 预算紧张，**优先**精简措辞、缩具体例子 / 长引文 / 重复论证
- **禁止**合并整段、删除小节、跳过命令 / 参数 / 错误码 / 表格行
- 该小节内容确实少时（如某些章节确实短），**显式**写"原文未明确"占位，而不是省略整节
- 整本 token 预算超限时：先输出所有章节骨架 + 前 N 章详细内容，剩余章节占位"原文已识别但因
  token 预算未展开——如需查看请使用 `--model gemini-3.1-pro-preview` 重跑"，**不要**直接截断

### 章节末尾的"故障排查 / FAQ / 更新日志"（manual 特有，**原样保留**）

- **故障排查 / 常见问题**章节（运维 / 安装类手册通常有）— 错误码表 + 排查步骤 + FAQ 完整保留
- **更新日志要点**（若手册含版本变更章节）— 1-2 句话点出本期主要变化

### 附录（如 PDF 含）

- **术语表 / Glossary** — 完整保留（术语 / 缩写 / 含义）
- **参考链接** — 文档里**明确给出**的链接保留（按"代码仓库 / 在线文档 / 视频 / 论坛"分组）；
  绝不凭自己知识补 URL；文档完全未给链接时整段省略
````
