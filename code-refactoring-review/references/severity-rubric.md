# 严重度判定细则(SSOT)

4 级严重度,本文件是唯一权威定义。SKILL.md 和 report-template.md 都引用本文件,不重抄。

## Blocker

**定义**: 重构后会改变行为 / 引入严重设计缺陷 / 必须立即修复。

**判定标准**:

- 重构后会改变行为(算法替换未测 / 边界条件丢失)
- 引入循环依赖
- 安全漏洞暴露(unvalidated input / SQL injection)
- 性能回归 > 20%
- 公共 API 破坏性变更未声明

**反例**(不该升 Blocker):

- 仅是命名不规范 → Major / Minor
- 仅是性能轻微下降 < 5% → Major

## Major

**定义**: 明显代码味,不修后期后果高。

**判定标准**:

- 函数 / 方法 > 50 行
- 重复代码块 > 5 行,重复 ≥ 3 处
- God Object(类承担 ≥ 3 个独立职责)
- Long Parameter List(> 5 个参数 / 参数簇 ≥ 3 个)
- 模块边界严重模糊(职责交叉)
- 复杂嵌套(条件嵌套 ≥ 3 层 / 循环嵌套 ≥ 2 层)
- 重要业务逻辑无测试覆盖
- 共享可变状态(模块级变量 / 全局状态)

**反例**:

- 仅是 1-2 行重复 → Minor
- 仅是函数稍长(30-50 行) → Minor

## Minor

**定义**: 局部改进 / 优化,值得做但非紧急。

**判定标准**:

- 命名不直观但能猜到(同义 / 缩写)
- 局部重复(< 3 处 / 每处 < 5 行)
- 轻微耦合方向问题(单向循环引用)
- 函数稍长(30-50 行)
- 缺少单元测试但非关键路径
- 多余抽象(1 层包装,行为一致)

**反例**:

- 函数 > 50 行 → Major
- 业务关键路径无测试 → Major

## Nitpick

**定义**: 风格 / 注释 / 可选项。

**判定标准**:

- 命名风格不一致(snake_case / camelCase 混用)
- 注释缺失(代码自解释) / 注释冗余(代码已经清楚)
- import 顺序 / 空行风格
- 函数 / 类文档字符串缺失(但行为清晰)
- TODO / FIXME 残留

**反例**:

- 命名与业务概念不符 → Minor / Major
- 误导性注释(注释说 A 但代码做 B) → Major

## 升级 / 降级规则

- 业务关键路径(支付 / 安全 / 权限)代码味可升一级(如 Minor → Major)
- 测试代码可降一级(测试代码冗余是常态)
- 文档字符串缺失若影响 IDE 提示可升 Nitpick → Minor

## 严重度与服务工具的对应

工具检测项的默认严重度(可在报告里调):

| 工具检测 | 默认严重度 |
| --- | --- |
| ruff unused import | Nitpick |
| ruff 重复代码 | Minor(按规模升 Major) |
| mypy 类型错误 | Minor(关键函数升 Major) |
| bandit 安全 | Blocker |
| radon 复杂度 > 10 | Minor;> 20 Major |
| vulture 死代码 | Nitpick |
