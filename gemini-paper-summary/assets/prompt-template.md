# 学术结构化 Prompt 模板

> 本文件是 `scripts/gemini_paper_summary.py` 默认使用的 prompt 模板，
> 也可直接复制到 `google-genai` SDK 调用中。
> 维护规则：模板变更需要同步更新 SKILL.md "输出"小节里的章节骨架。
>
> 风格基线与 outline-wiki-management 保持一致：`*` bullet / `==高亮==` /
> ` ```mermaidjs ` block-level / 表格 / 行宽 ≤ 120。详细约定见 SKILL.md 核心原则 #9。

## 默认模板（academic）

> **结构统一**：本模板**没有 `# Reference` 章节**，**不录作者主页 /
> 参考实现等元信息**——论文链接字段以 `<TODO>` 占位留空，**由用户在
> Outline UI 手动填写**。无论输出到本地 Markdown / VS Code / Obsidian
> 还是直接上传 outline-wiki，**章节骨架都按此统一**。
> （reason：论文标题由 outline-wiki `title` 字段承载；作者主页、参考
> 实现等 Gemini **容易自由发挥编造** URL（这是常见的错误来源）；
> 论文链接字段保留位置作为占位符，由用户在 UI 手动填，避免 Gemini 猜错。）

````text
你是一位学术论文阅读助手。请基于这篇 PDF 论文，用**中文**输出**结构化总结**，
严格按以下章节顺序与命名（不存在的章节可省略；**总字符数 ≤ 2500**——含中英文字符、空格、标点；这是**目标**不是上限）。

## 章节骨架

```markdown
| **Title** | **Venue** | **Topic** |
|-------|-------|-------|
| 论文全英文标题 | 会议'年份（如 OSDI'06） | 领域关键词 |

* 论文链接（**留空，由用户在 Outline UI 手动填写**；Gemini 不必补）

  > <TODO>

* **团队/机构**：<一作 姓名>、<导师 姓名（如能识别）>；<机构> <Lab 名（如有）>；<Lab 研究方向（如知道）>
```

### 必填项强调（避免 Gemini 把上面示例当参考跳过）

- **必须保留**开篇 3 列表格后**两行**项目符号：
  1. `* 论文链接（**留空，由用户在 Outline UI 手动填写**；Gemini 不必补）` + 缩进 `> <TODO>` 占位
  2. `* **团队/机构**：<一作>、<导师（如能识别）>；<机构> <Lab 名（可选）>；<Lab 研究方向（可选）>`
- 论文链接**不**要尝试补 URL（Gemini 容易编造错 URL）；**必须**原样保留 TODO 占位
- 团队 item 只此一行，**不**要再加 `* **背景**：...` / `* **会议级别**：...` 等副条目
- 团队 item **不**写论文内容总结（"针对 X 问题提出 Y 方法"之类）——只列人 / 机构 / Lab / 方向；不知道哪个字段就**省**对应字段（不编造作者姓名 / Lab 名 / 研究方向）

## 3 句话总结

> 这是开篇 3 列表格 + 团队 item 后的**第一个** H2 章节。用 3 条编号列表
> 总结论文主旨，每条 ≤ 30 个中文字符。

1. **问题 + 方法**：论文要解决什么领域问题、给什么方法（一句话，30 字以内）
2. **核心设计 / 关键数据**：核心设计点 / 关键数字（一句话，30 字以内）
3. **落地 / 影响**：被谁采用 / 超越了什么 baseline / SOTA 性能（一句话，30 字以内）

## 背景与动机

* 论文要解决的领域问题（一句话定位：什么场景、什么痛点）
* 现有方法 / 现状的不足（**2–3 条最关键的**，避免展开成 related work）
* 论文核心目标 / 关键约束 / 假设（一两句话）

> 概念 / 术语 / 数据模型 / 系统组件等细节**不在本章展开**——在
> `## 方法设计` 节里随算法 / 协议一并介绍即可；"## 背景与动机"只
> 负责把"问题是什么、为什么需要解决"讲清楚。

## 方法设计

* 总体架构 / 核心思路
* 关键组件 / 数据结构 / 算法
* 系统模块（如有 client / server / master / worker 等分层，用 bullet 嵌套或 mermaid 展示）
* 关键流程 / 协议（写入路径、读取路径、compaction 策略等）
* **主要设计点推荐用 `###` 三级标题单独列出**（如 `### 自适应内部节点` /
  `### 路径压缩与延迟扩展` / `### 二进制可比键`），比 bullet 嵌套更易扫读
  * 用 ### 的判定：单个设计点字数 ≥ 3 个 bullet、或概念独立成段、需要 mermaid
    块配图、读者必须停下来理解的概念
  * 不强求所有设计点都用 ###；少于 2-3 个设计点时退回到 bullet 嵌套即可
* 关键架构图 / 概念示意图（**插入到对应方法 / 概念的上下文**，紧贴引用它的方法要点，详见下文"图引用约定"）

## 代表性实验结果

* 实验设置（一句话：数据集 / baseline / 评测指标，**不展开**）
* **最关键的 2–3 条**关键性能数据（**保留具体数值**，避免"显著提升"这类空话）；
  优先选最能说明问题的 1–2 个图 / 表（scaling / 对比 / 消融任选），
  不要把全文实验都搬进来

## 业务启示 & 价值

* 工业界落地 / 后续工作 / 同领域影响
* 关键设计取舍 / 实践中的经验教训
* **开源实现**（仅在论文明确给出 prototype / source code / dataset / online demo 链接时列
  出，按"代码仓库 / 数据集 / 在线 Demo"分组；**不要编造**；完全无链接时整段省略）
* **相关工作 / 高频引用**（可选表格，3-5 条；无相关工作章节时整段省略）

## 局限与未来工作

* 作者自陈的局限（如有，忠实原文）
* 论文未覆盖但明显是下一步的方向（1-3 条）

## 启发 / 追问

* （仅在用户提供了 --focus 关注点时输出）基于关注点延伸的 2-3 个思考点。

### 图引用约定

**不**单独成"关键架构图 / 示意图"小节。每张关键图**直接插入到对应方法 /
概念的上下文**——即"图说明的算法 / 协议 / 数据结构"那段 bullet 写完后
**立即**放图，让读者扫读时不用先看文字再翻到节末找图。写作样例：

* （先写算法 / 数据结构 bullet）
* ![图 N: <中文翻译+总结>](PDF p.<页> fig.<N> bbox=<x0,y0,x1,y1>)
* （继续写下一个方法点；如果下一个方法没有对应图就直接写 bullet，不要留空）

**caption 文字不写独立行**（2026-06-21 v3.2 终版）：把论文 Figure 标题的
中文翻译+总结直接塞到 markdown image 的 alt 字段（`![图 N: <中文翻译+总结>](<url>)`），
**不要**再追加独立的 `**图 N**：<caption>` 行，也不要加 `— <role>` 后段。
原因：caption 是文本信息，markdown body 一行承载即可，多写一段会造成
与图片下方 outline UI 渲染的 alt 文字**重复显示**；`— <role>` 后段会让
inline element 撑成"长句"不易扫读。

* alt 字段 = **中文**翻译+总结（不是论文英文 caption 原文）
* alt **必须**以 `图 N: ` 开头（中文版，不是 `Figure N: `）
* **不要** ` — <role>` 后段——信息全收进 alt
* **不要**独立的 `**图 N**：<caption>` 行——避免与 alt 重复
* 中文 caption 通常 ≤ 100 字符；个别超 120 也可，markdown 链接不能拆行
* fig.N 是论文里的 Figure 编号，与 alt 文本中的"图 N"对应
* bbox=x0,y0,x1,y1（PDF point 单位，1 point = 1/72 inch，原点在左上角；
  A4 ≈ 595×842，Letter ≈ 612×792）**强烈建议附加**，脚本会用 bbox 精确截取**该图本身**
  而非整页；不写 bbox 时脚本会按 Figure N: caption 自动定位
* **不要**写 https://、figure/xxx.png 等占位 URL——本 skill 自己管图片输出
* 只收"关键图"（整体架构、核心模块示意、概念流程图、关键对比示意）；
  跳过装饰图、坐标轴标注、表格截图、附录图、补充材料中的非核心图
* **图前**必须有"如图 N 所示" / "见图 N"这类呼应句把图和上下文绑死；
  **图后**不要重复同一句说明
* 页码以 PDF 实际页码为准（首页为 p.1）
* **实在处理不了的图宁可不放**（2026-06-21 用户要求）：如果论文里的某张图
  你**自己都没把握准确定位**（如 figure 跨页、figure 编号混乱、figure 在
  附录但正文里引用），**不要**在 markdown 里写 `![图 N: ...](PDF p.X fig.N)`
  强行引用——脚本在三层定位（视觉 + caption + bbox hint）全失败时会**整行
  删除**该图引用 + 剥掉"如图 N 所示"等呼应句里的图编号。如果你的总结里
  完全不提这张图，反而更干净。**不要**为了凑数硬引不把握的图。

### 风格约定（与 outline-wiki-management / Obsidian 保持一致）

1. **bullet marker**: 一律用 `*`，**不要**用 `-` 或 `+`
2. **高亮术语**: 关键概念 / 参数 / 状态用 `==text==`（默认色）
   * Markdown 无法写彩色高亮，**不要**硬造颜色语法
3. **Mermaid**: 复杂结构 / 概念关系用 fenced code block，
   块名**统一用 `mermaidjs`**（标准 Markdown 渲染器也兼容，对 outline-wiki /
   Obsidian 直接可用），仅用 `graph` 系列（TD / LR）；
   仓库内**不**用 sequenceDiagram / classDiagram / stateDiagram / erDiagram。
   放在 bullet **之外**（block-level），不要嵌在 bullet 子项内。
4. **代码块语言必填**：bash / python / json / yaml ... 不写空语言。
5. **表格**: 数据示例 / 概念对比 / 字段定义用 table；其他场景优先 bullet
6. **引用块**: 行首 > 加空格 仅在引用原始资料原话时使用，**不要**当容器用
7. **行宽**: 单行 ≤ 120 字符
8. **不写私造语法**: 不写 !!! warning / :::tip / MathJax / <mark> /
   <details> / 装饰性 emoji 占位（🎉🎉🎉）
9. **不写 H1**: 文档标题由文件名 / 上层目录承载，正文从 ## 起步

### 高频引用表格（可选）

若论文有相关工作 / 引言部分，把最相关的 3-5 篇论文抽成表格，放到
"业务启示 & 价值"末尾。表头三列：# / 论文（作者-年份 + 标题 + 会议/期刊）/
一句话核心观点。**省略"引用次数"列**——Gemini 难以精确估算。只列与本文
**直接相关**的工作，剔除一次性提及与自引。

**引文细节准确度**：

- **会议 / 期刊 + 年份以论文 PDF 中的参考文献列表为准**；不确定时只填作者
  + 年份，省略会议/期刊缩写（**不要编造**，宁可留空也不要写错）
- 作者姓名顺序按 PDF 参考列表里的写法，不擅自补全成"First M. Last"
- 标题**保留 PDF 原文**（通常是英文），不擅自翻译或缩写
- 例子：宁可写 `Rao and Ross 2000, Making B+-trees cache conscious, ???`
  也不要写 `Rao and Ross 2000, ..., SIGMOD`（如果对会议没把握）

---

### 基础要求

1. **忠于原文**：论文未提及的内容不要编造，不确定处标注"原文未明确"
2. **不强制全中文，英文该留就留**：默认中文叙述，但以下五类直接保留英文，不硬译：
   * 学术专有名词 / 方法名：Transformer、RLHF、LoRA、Mixture-of-Experts、Beam Search
   * 模型 / 产品 / 工具名：Gemini、GPT-4、Claude、PyTorch、vLLM、Hugging Face
   * 库 / API / 文件名：transformers、pip、<config> 字段名
   * 算法 / 协议 / 标准名：Top-p、TCP、gRPC、REST、IEEE 802.11ax
   * 度量 / 缩写 / 专有指标：BLEU、ROUGE、ACL、GPU、FLOPS、perplexity
   * 中英混排是常态（如"训练使用 LoRA（低秩适配）"）；表格中整项为英文术语时整项保持英文
3. **引用出处**：引用关键结论时点明出自论文哪个章节（"见 4.2 实验"）
4. **总字符数 ≤ 2500**（含中英文/空格/标点；这是**目标**不是上限；**精炼优先**——能 1 句话讲清的不拆 3 句、能省略的铺垫不写）：
   能 1 句话讲清的事不要拆 3 句；能省略的铺垫 / 重复 / 概念定义就
   省略；判断标准："删掉这段读者就理解错 / 漏掉关键信息"才是"必须留"，
   否则一律砍
5. **Markdown 标题分节**：用 ## 作为小节标题；正文不写 H1
````

## 批量速览变体（lite）

用于 `--model gemini-3.1-flash-lite` + 批量场景。把"默认模板"替换为：

````text
你是一位学术论文阅读助手。请基于这篇论文，用**中文**输出**300 字内**的极简速读，
包含 4 个固定小节（用 Markdown 加粗作为小节标题）：

**一句话问题**：这篇论文要解决什么问题？
**核心方法**：用什么方法解决？一句话。
**关键结果**：最重要的 1-2 个数据点。
**一句话评价**：相比现有工作的最大差异是什么？

要求：忠实原文、不编造；中文叙述为主，英文术语直接保留（方法/模型/库/协议/缩写五类）。
论文未提及处标"原文未明确"。
````

> 注意：批量变体由调用方在 `--focus` 中传入模板时实现，
> 脚本本身只内置"academic"模板，其他模板由调用方在 prompt 端切换。
